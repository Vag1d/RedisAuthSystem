# Система авторизации с JWT и Redis (белый/черный списки)

## Описание

Реализована система аутентификации и авторизации на основе JWT-токенов с использованием Redis для хранения **чёрного списка** access-токенов и активных refresh-токенов. Поддерживаются две роли: `user` и `admin`. Каждая роль имеет доступ к общему контенту, а также к эксклюзивному контенту только своей роли.



## Чёрный список токенов
При логауте access-токен добавляется в Redis с ключом blacklist: token и TTL, равным оставшемуся времени жизни токена.

При каждом запросе к защищённому эндпоинту middleware проверяет наличие токена в чёрном списке. Если есть – доступ запрещён.

## Refresh-токены
Refresh-токен хранится в Redis с ключом refresh: username.

При обновлении токенов старый refresh удаляется, создаётся новая пара.

Это обеспечивает ротацию и возможность отозвать все сессии пользователя (удалив ключ).

## Архитектура

- **FastAPI** – асинхронный веб-фреймворк.
- **Redis** – хранилище чёрного списка и refresh-токенов.
- **JWT** – создание и верификация токенов.
- **Passlib + bcrypt** – хеширование паролей.

## Почему REST API?

Выбран **REST API** как стандартный, масштабируемый и простой способ взаимодействия с бэкендом. Он:
- Позволяет легко интегрироваться с любыми клиентами (SPA, мобильные приложения).
- Использует stateless-подход (кроме Redis, который служит вспомогательным хранилищем).
- Предоставляет автоматическую документацию OpenAPI (доступна по `/docs`).

## Обнаружение и защита от утечек токенов

### Возможные утечки
- Перехват токена по сети (при отсутствии HTTPS).
- XSS-атаки (если токен хранится в localStorage).
- Компрометация клиентского устройства.
- Утечка через логи, заголовки.

### Как определить кражу
- Отслеживание аномального поведения (смена IP, подозрительная временная активность).
- Привязка токена к fingerprint устройства (User-Agent + IP) – в данном решении не реализовано, но может быть добавлено.
- Короткое время жизни access-токена (15 минут) снижает риск.

### Способы защиты (реализованные и рекомендуемые)
- **HTTPS** – обязателен.
- **Access-токен живёт 15 минут**, refresh – 7 дней.
- **Чёрный список Redis** – при логауте токен немедленно отзывается.
- **Refresh-токены хранятся только в Redis** (не передаются повторно без необходимости).
- **Ротация refresh-токенов** – при каждом обновлении создаётся новый.
- Рекомендуется также использовать **HttpOnly Secure Cookies** для хранения токенов (защита от XSS).

## Запуск проекта

### 1. Запуск с Docker 

```bash
git clone https://github.com/Vag1d/RedisAuthSystem.git
cd RedisAuthSystem
docker-compose up --build
```

### 2. Запуск локально

```bash
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

##### Установать и запустить redis 

```bash
sudo apt update
sudo apt install redis-server
sudo systemctl start redis
redis-cli ping
```
